Exception Handling and Error Responses

In real-world applications, things will go wrong. In this lesson, we set up centralized exception handling for the FinBank API.

Centralized exception handling

A consistent error layer is a major quality marker. It also makes AI-assisted changes safer: you don’t have ad-hoc error formats scattered across controllers. 

import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.ExceptionHandler;
import org.springframework.web.bind.annotation.RestControllerAdvice;

@RestControllerAdvice
final class ApiExceptionHandler {
  @ExceptionHandler(NotFoundException.class)
  ResponseEntity<ApiError> notFound(NotFoundException ex) {
    return ResponseEntity.status(HttpStatus.NOT_FOUND)
        .body(ApiError.notFound(ex.getMessage()));
  }
}

Keep ApiError stable: it’s part of your public contract.

AI Review Checklist (Exceptions)

  • Are exceptions mapped to the right status codes?
  • Is error shape consistent across endpoints?
  • Any leaks of internal details?
  • Are logs/traces correlated (traceId)?

Quest

  1. Create 3 custom exceptions: not found, conflict, validation.

  3. Ask AI to add more mappings and review for status consistency.
  4. Add tests for error shape and status codes.
