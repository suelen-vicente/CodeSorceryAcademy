Part 11

Using DTOs

Until now, you might be validating directly on an entity. It works at first. But in real systems, mixing API formats with persistence entities causes long-term pain.

← Part 10 Back to Timeline

Why DTOs matter

Stability

Entities evolve with DB. API contracts should evolve carefully.

Security

Entities often contain internal fields you should never expose.

Clarity

Request/response formats become explicit and documented.

AI safety

AI-generated code tends to “just return the entity”. DTOs prevent that leak.

Example: Entity vs DTO

// Entity (persistence shape)
final class CustomerEntity {
  String id;
  String name;
  String email;
  String internalRiskFlag; // never expose
}

// DTOs (API contract)
record CreateCustomerRequest(String name, String email) {}
record CustomerResponse(String id, String name, String email) {}

Keep mapping explicit. If your system grows, consider mapping tools — but don’t hide logic in “magic mapping”.

AI Review Checklist (DTOs)

  • Are controllers returning DTOs (not entities)?
  • Is mapping code tested for edge cases?
  • Any accidental exposure of internal fields?
  • Is validation placed on request DTOs?

Final Quest

  1. Refactor one endpoint: entity-in, entity-out → DTO-in, DTO-out.
  2. Add a new internal entity field and confirm the API contract stays stable.
  3. Ask AI to refactor another endpoint and review for leaks and boundary mistakes.
