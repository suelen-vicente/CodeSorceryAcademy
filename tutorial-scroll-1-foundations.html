Feitiço Base: “Invariantes ou Caos”

A maioria do código ruim (incluindo o gerado por IA) falha num ponto simples: não declara as regras do domínio. Declare invariantes. Faça o estado do objeto impossível de ficar inválido. 

import java.math.BigDecimal;
import java.util.Objects;

final class Money {
  private final BigDecimal amount;
  private final String currency; // "BRL", "USD"

  Money(BigDecimal amount, String currency) {
    Objects.requireNonNull(amount, "amount");
    Objects.requireNonNull(currency, "currency");
    if (amount.scale() > 2) throw new IllegalArgumentException("scale > 2");
    if (amount.signum() < 0) throw new IllegalArgumentException("negative money");
    if (currency.length() != 3) throw new IllegalArgumentException("invalid currency");
    this.amount = amount;
    this.currency = currency;
  }

  Money add(Money other) {
    if (!this.currency.equals(other.currency)) throw new IllegalArgumentException("currency mismatch");
    return new Money(this.amount.add(other.amount), this.currency);
  }
}

Pergunta de senior: “Qual estado inválido esse tipo impede?” Se a resposta for “nenhum”, a complexidade vai migrar (e explodir) em outro lugar.

Armadilhas comuns (principalmente com IA)

“Parece certo” não é certo

Código de IA costuma ser plausível. Você valida com testes, invariantes e exemplos extremos (edge cases).

Complexidade escondida

Streams lindos com custo O(n²), conversões desnecessárias, autoboxing e alocações em loop.

Nulls e contratos frágeis

Defina: aceita null? retorna Optional? lança exceção? Documente e teste. Sem isso, seu sistema vira “adivinhação”.

Ritual prático (30–60 min)

  1. Escolha uma entidade do mundo real (ex.: Conta, Pedido, Sessão).
  2. Escreva 5 invariantes (o que nunca pode ser falso).
  3. Modele o tipo em Java para impedir estados inválidos.
  4. Peça para a IA sugerir uma implementação e faça um review: invariantes estão garantidas? Onde pode quebrar?
  5. Crie 10 testes: 6 casos felizes + 4 “ataques” (inputs estranhos, limites, combinações).

Prompt bom (use como familiar)

“Implemente o tipo X garantindo estas invariantes: … Mostre exemplos de testes de unidade cobrindo edge cases. Aponte trade-offs e riscos.”
