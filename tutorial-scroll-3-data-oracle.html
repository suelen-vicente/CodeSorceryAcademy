Dados & Oracle: verdade, custo e concorrência

Banco é onde a realidade mora. E Oracle é um reino com regras próprias: execução, planos, locks e custos que não perdoam. Aqui você aprende a modelar para clareza e performance — e a revisar JPA com olhos de produção.

Princípio do pergaminho

Você não “usa banco”. Você negocia com ele. Cada consulta tem custo. Cada índice tem preço. Cada lock tem consequência. A competência aparece quando você consegue responder: “O que isso faz com concorrência, cardinalidade e plano de execução?”

Verdade

Constraints, chaves, integridade. O esquema é parte do domínio.

Custo

I/O, CPU, índices, estatísticas, joins. Otimizador não faz milagre.

Concorrência

Transações, locks, isolamento, idempotência. O bug mora no “ao mesmo tempo”.

Modelagem que evita feitiço reverso

“Tabela por entidade” é o começo, não o fim. Pense em invariantes e em como o banco protege o sistema contra bugs.

Checklist (schema como contrato)

  • PKs estáveis (evite “natural key” quando ela muda na vida real).
  • FKs e constraints onde o banco pode garantir invariantes.
  • Colunas com semântica clara (evite “status_code_1”, “flag2”).
  • Índices guiados por queries reais (não por esperança).

Oracle: índices, planos e estatísticas

Performance em Oracle não é adivinhação: é plano. Se você não viu o plano, você não sabe. 

-- Exemplo (didático): query típica de API
SELECT o.id, o.customer_id, o.status, o.created_at
FROM orders o
WHERE o.customer_id = :customerId
  AND o.created_at >= :from
ORDER BY o.created_at DESC
FETCH FIRST :limit ROWS ONLY;

Índice que conversa com a query

Um índice composto em customer_id, created_at pode reduzir I/O e ajudar o ORDER BY. Mas lembre: cada índice custa em INSERT/UPDATE.

Estatísticas importam

Sem estatísticas boas, o otimizador pode escolher planos ruins. Em produção, o “de repente ficou lento” muitas vezes é isso.

Pergunta de review (IA + JPA)

“Que SQL isso gera? Qual o plano provável? Ele escala com cardinalidade ou só com dados de teste?”

JPA/Hibernate: a magia e o preço

ORM acelera produtividade — e cria armadilhas silenciosas. Você precisa reconhecer padrões que detonam performance ou consistência.

N+1 (o clássico)

Se você não controla fetch, a IA vai “resolver” com loops e explode o banco com queries.

Paginação e ORDER BY

Pagination sem índice compatível vira full scan. Em Oracle, conheça bem as estratégias (OFFSET/FETCH) e o custo real.

Transação “grande demais”

Transação longa vira lock, lock vira fila, fila vira incidente. Quebre o fluxo, use outbox, replaneje.

Quest: do endpoint ao plano de execução

  1. Escolha um endpoint real (ex.: listar pedidos com filtros e ordenação).
  2. Escreva a query SQL “ideal” primeiro (com parâmetros e limites).
  3. Defina 1–2 índices candidatos e justifique (custo de escrita vs leitura).
  4. Implemente com JPA (ou JDBC) e peça para a IA sugerir melhorias.
  5. Revise a sugestão com foco em: N+1, paginação, transação, locks e SQL gerado.

Se você quiser subir o nível

Adicione um caso de concorrência: duas requisições atualizando o mesmo pedido ao mesmo tempo. Decida estratégia: optimistic lock (version), pessimistic lock, ou fila/evento.

Agora que os dados estão sob controle, é hora de colocar magia no mundo: build, deploy e ambientes.
