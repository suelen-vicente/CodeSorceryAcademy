Introduction to RESTful APIs

Now that we’ve set up our development environment, explored Java fundamentals, and handled errors, it’s time to understand the principles behind RESTful APIs.

What is REST?

REST (Representational State Transfer) is an architectural style for building APIs. It uses HTTP methods to operate on resources, which are typically represented in JSON.

In a bank example, a resource could be:

  • A Customer
  • A Bank Account
  • A Transaction

Resources and Endpoints

A resource is identified by a URL. For example: 

GET /api/customers/123

This endpoint retrieves the customer with ID 123.

HTTP Methods in REST

In REST, different HTTP methods indicate what action you want to perform:

Method Purpose Example in FinBank
GET Retrieve data GET /api/accounts/456 → Get account details
POST Create new data POST /api/accounts → Create a new account
PUT Update an existing resource PUT /api/accounts/456 → Replace account details
PATCH Partially update a resource PATCH /api/accounts/456 → Update account balance only
DELETE Remove a resource DELETE /api/accounts/456 → Close the account

HTTP Status Codes

When a REST API responds, it includes a status code to indicate the result. There’s a variety of codes that can be used (like 422, 403, etc.), but here are common ones:

Code Meaning Example in FinBank
200 OK Request succeeded Account details retrieved
201 Created Resource created New account successfully opened
400 Bad Request Invalid input Transfer request missing amount
401 Unauthorized Authentication failed Missing or invalid API token
404 Not Found Resource not found Account ID doesn’t exist
500 Internal Server Error Server failure Unexpected error during transfer

Usually, 2xx expresses success. 4xx means something blocked the operation. 5xx is reserved for server and generic errors.

JSON – The Standard API Data Format

Most modern APIs exchange data in JSON.

Example – JSON response from GET /api/accounts/456:

{
  "accountNumber": "456",
  "accountType": "Checking",
  "balance": 1200.50,
  "currency": "USD"
}

Designing RESTful Endpoints

REST APIs should be intuitive and resource-based, not action-based. You create endpoints to manipulate resources. The actions are defined by the HTTP method you use.

Not recommended

POST /api/createNewAccount

Recommended

POST /api/accounts

Example:

GET /api/accounts/456/transactions

Returns all transactions for account 456.

REST Best Practices

  • Use nouns, not verbs in URLs (/customers, not /getCustomers)
  • Use plural resource names (/accounts, not /account)
  • Use HTTP status codes correctly
  • Version your API (/api/v1/accounts)
  • Support filtering and pagination for large datasets

Example:

GET /api/transactions?minAmount=1000&page=2&size=20
