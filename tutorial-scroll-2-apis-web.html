APIs & Web: contratos, limites e erros

API não é controller bonito. API é contrato, compatibilidade, segurança e previsibilidade. É aqui que você para de “fazer funcionar” e começa a construir sistemas confiáveis.

Mapa do pergaminho

HTTP como linguagem

Métodos, status codes, caching, headers. Você usa o protocolo a seu favor.

Versionamento, breaking changes, DTOs, validação, e o custo de mudar em produção.

Feitiço Base: resposta de erro como contrato

Uma API madura trata erros como produto. Erro precisa ser previsível (máquinas) e explicável (humanos). Isso te dá poder para revisar código gerado por IA: “O contrato do erro ficou consistente em todas as rotas?” 

{
  "error": "validation_error",
  "message": "Campo 'email' inválido.",
  "path": "/v1/users",
  "traceId": "01HZX...9Q",
  "details": [
    { "field": "email", "reason": "must be a well-formed email address" }
  ]
}

Checklist de review (o que a IA costuma esquecer)

  • Status code correto (400/401/403/404/409/422/429/5xx).
  • Mesmo formato em todos os endpoints (sem surpresas).
  • TraceId correlacionável (logs/traces).
  • Sem vazamento de detalhes sensíveis (stack trace, SQL, nomes internos).

Spring Boot: o mínimo que vira máximo em produção

Em APIs reais, você ganha o jogo nos detalhes: validação, boundaries, logs, observabilidade e testes. Controller é a borda. O domínio é onde a verdade mora. 

import jakarta.validation.constraints.Email;
import jakarta.validation.constraints.NotBlank;
import org.springframework.http.HttpStatus;
import org.springframework.web.bind.annotation.*;

record CreateUserRequest(@NotBlank String name, @Email String email) {}
record CreateUserResponse(String id) {}

@RestController
@RequestMapping("/v1/users")
final class UsersController {
  private final UsersService usersService;

  UsersController(UsersService usersService) {
    this.usersService = usersService;
  }

  @PostMapping
  @ResponseStatus(HttpStatus.CREATED)
  CreateUserResponse create(@RequestBody @jakarta.validation.Valid CreateUserRequest req) {
    return new CreateUserResponse(usersService.create(req.name(), req.email()));
  }
}

Nota de senior: DTOs são contrato. Não exponha entidade. E não deixe validação “escorrer” pro domínio sem uma tradução explícita de erro.

Idempotência: o feitiço anti-duplicação

Clientes retryam. Proxies retryam. Seus workers retryam. Sem idempotência você cria cobranças duplicadas, pedidos duplicados e bugs “fantasmas”.

Prática recomendada

  • POST sensível recebe Idempotency-Key.
  • Você registra o resultado por key (com TTL) e retorna o mesmo response em retries.
  • No banco: key única + transação (sem race conditions).

Pergunta que sobe carreira

“Esse endpoint é safe contra retry? E contra duas requisições concorrentes do mesmo usuário?”

Quests (com e sem IA)

  1. Desenhe um endpoint de criação de pedido com idempotência e erros padronizados.
  2. Escreva uma especificação curta (contrato): request/response, erros, status codes, headers.
  3. Peça para a IA implementar “controller + service + testes”.
  4. Faça review com foco em: validação, erros, idempotência, e boundary (controller vs domínio).
  5. Adicione 5 testes de contrato (ex.: error shape, status code, field validation).
