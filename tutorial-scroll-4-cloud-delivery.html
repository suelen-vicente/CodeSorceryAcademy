Cloud & Entrega: rituais de deploy sem medo

O código não vale nada até rodar com segurança em produção. Aqui você aprende a transformar API em serviço: build reprodutível, configuração correta, deploy previsível e rollback rápido.

O que muda quando você “vira produção”

Reprodutibilidade

Mesma build, mesmo artefato, mesmo resultado. “Na minha máquina” não entra.

Configuração como contrato

Ambientes, secrets, flags, timeouts. Config errado é bug de produção.

Observabilidade

Logs, métricas e traces. Se você não mede, você não controla.

Rollback e segurança

Mudanças pequenas, reversíveis e com blast radius reduzido.

Docker: o círculo de invocação

Container não é “só empacotar”. É declarar dependências, reduzir superfície e garantir um runtime previsível. 

# Exemplo didático (multi-stage) para Spring Boot
FROM eclipse-temurin:21-jdk AS build
WORKDIR /app
COPY . .
RUN ./mvnw -q -DskipTests package

FROM eclipse-temurin:21-jre
WORKDIR /app
COPY --from=build /app/target/app.jar /app/app.jar
EXPOSE 8080
ENTRYPOINT ["java","-jar","/app/app.jar"]

Review de IA (Dockerfile)

  • Evitar rodar como root quando possível.
  • Imagem base consistente (JRE vs JDK) e versão fixa.
  • Build stage separado (reduz tamanho e risco).
  • Não vazar secrets em layers.

CI/CD: o ritual que protege o time de si mesmo

Pipeline bom não é o mais complexo. É o que torna o caminho seguro e barato: testes rápidos, build determinístico, deploy previsível.

Gate 1

Lint/format + unit tests rápidos.

Gate 2

Build do artefato + testes de integração essenciais.

Gate 3

Deploy com estratégia (blue/green/canary) + smoke checks.

Pergunta de senior

“O que impede um deploy ruim de chegar em produção?” Se a resposta for “esperança”, você está devendo rituais.

Configuração e secrets: o grimório invisível

A mesma aplicação roda em dev/staging/prod. O que muda é configuração. Faça isso explícito e auditável.

  • Use profiles com propósito (ex.: local, staging, prod).
  • Timeouts e retries configuráveis por ambiente.
  • Secrets fora do repo (vault/secret manager), com rotação.
  • Feature flags com owner e data de expiração (flag sem dono vira dívida).

Quest: deploy com rollback em 20 minutos

  1. Escolha um serviço Spring Boot simples (ou crie um hello API).
  2. Crie um Dockerfile multi-stage e faça build local.
  3. Defina 3 configs por ambiente (ex.: log level, timeout, feature flag).
  4. Simule um deploy com duas versões e uma “chave de rollback” (tag/versão).
  5. Peça para a IA sugerir melhorias no pipeline e faça review com foco em segurança e reprodutibilidade.

Último pergaminho: confiabilidade. O lugar onde sistemas deixam de ser “funciona” e viram “aguenta”.
