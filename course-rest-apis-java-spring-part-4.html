Error Handling & Validation

When building APIs, it’s critical to handle errors gracefully and validate all incoming data. Poor error handling can lead to confusing API responses and security issues.

API Errors: What Clients Should See

In an API, a bad request shouldn’t cause the application to crash or return raw Java stack traces. Instead, we return a clear, client-friendly JSON error.

Example of a bad practice (stack trace to client):

{
  "error": "NullPointerException at TransactionService.java:42"
}

Example of a good practice:

{
  "status": "error",
  "message": "Account not found",
  "errorCode": 404
}

Exception Handling in Java

Java uses the try, catch, and finally keywords for handling exceptions.

Example – Catching an exception when transferring funds:

public void transferFunds(String fromAccount, String toAccount, double amount) {
  try {
    // Attempt transfer
    processTransfer(fromAccount, toAccount, amount);
  } catch (InsufficientFundsException e) {
    System.out.println("Error: " + e.getMessage());
  } finally {
    System.out.println("Transaction log updated");
  }
}

Creating Custom Exceptions

Custom exceptions make API responses more meaningful. 

public class AccountNotFoundException extends RuntimeException {
  public AccountNotFoundException(String accountNumber) {
    super("Account not found: " + accountNumber);
  }
}

Usage in a service:

public Account getAccount(String accountNumber) {
  return accountRepository.findById(accountNumber)
      .orElseThrow(() -> new AccountNotFoundException(accountNumber));
}

Global Exception Handling in Spring Boot

Instead of handling exceptions in every controller, we can use @ControllerAdvice to centralize them. 

import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.ControllerAdvice;
import org.springframework.web.bind.annotation.ExceptionHandler;

@ControllerAdvice
public class ApiExceptionHandler {

  @ExceptionHandler(AccountNotFoundException.class)
  public ResponseEntity<ErrorResponse> handleAccountNotFound(AccountNotFoundException ex) {
    ErrorResponse error = new ErrorResponse("error", ex.getMessage(), 404);
    return new ResponseEntity<>(error, HttpStatus.NOT_FOUND);
  }
}

ErrorResponse DTO:

public class ErrorResponse {
  private String status;
  private String message;
  private int errorCode;

  public ErrorResponse(String status, String message, int errorCode) {
    this.status = status;
    this.message = message;
    this.errorCode = errorCode;
  }
}

Input Validation with Bean Validation

Spring Boot integrates Bean Validation to automatically check incoming request data.

Example – DTO with validation:

import jakarta.validation.constraints.Min;
import jakarta.validation.constraints.NotBlank;

public class TransferRequest {
  @NotBlank(message = "From account is required")
  private String fromAccount;

  @NotBlank(message = "To account is required")
  private String toAccount;

  @Min(value = 1, message = "Amount must be greater than zero")
  private double amount;

  // getters and setters
}

Controller method:

import org.springframework.validation.annotation.Validated;
import org.springframework.web.bind.annotation.*;

@RestController
@RequestMapping("/api/transfers")
public class TransferController {

  @PostMapping
  public String transfer(@Validated @RequestBody TransferRequest request) {
    // process transfer
    return "Transfer successful";
  }
}

If a request fails validation, Spring automatically returns a 400 Bad Request with details.
